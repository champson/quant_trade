# Quant Trade 设计文档

## 1. 目标

Quant Trade 是统一的 A 股研究工作台，覆盖四类操作：

1. 每日收盘复盘；
2. ETF、微盘股及后续策略的信号生成与回测；
3. 日线和分钟数据的下载、导入、校验与查询；
4. 可复现的研究产物、报告和运行记录。

原来的 `etf/`、`market_breadth/` 独立目录已移除。业务能力统一放在
`src/quant_trade/`，运行入口统一为 `qt`，生成文件统一进入 `artifacts/`。

## 2. 分层结构

```text
CLI / Dashboard
       │
       ├── pipelines       每日收盘等编排任务
       ├── services        策略信号、回测和数据更新用例
       ├── strategies      只负责从标准数据产生目标权重或信号
       ├── backtest        撮合、绩效计算和报告生成
       ├── reports         市场复盘报告
       └── data
            ├── providers  Tushare / Baostock / AkShare
            ├── router     重试、熔断和数据源降级
            ├── importers  ZIP及本地分钟目录导入
            └── storage    Parquet + DuckDB目录
```

策略不直接调用数据源、不自行管理缓存，也不自行绘图。新增策略只需实现标准信号和目标
权重接口，由公共服务完成数据读取、回测、成本建模和报告输出。

## 3. 数据设计

### 3.1 在线数据

数据路由按配置的优先级访问 Tushare、Baostock 和 AkShare。网络请求使用有限次数重试、
退避、熔断和可选数据源降级。交易日历按日期覆盖缓存在 DuckDB；历史日期永久复用，
今天及未来日期按 `calendar_mutable_ttl_hours` 定期刷新，也可由服务层强制刷新。行情缓存
完整时，更新过程不需要联网。数据来源与请求结果同样记录在 DuckDB。

日线按资产类型和复权方式物理隔离，行情行同时保留 `asset_type`、`adjustment` 和
`source`：

```text
data/processed/daily/
  etf/
    adjustment=hfq/
      510300_SH.parquet
```

复权方式只允许 `none`、`qfq`、`hfq`。Provider 不能满足请求时必须降级或失败，标准化
结果的复权标签与请求不一致时拒绝写入；BaoStock 还校验数据源返回的 `adjustflag`，
Tushare 的股票和 ETF 后复权由未复权日线乘以逐日 `adj_factor`/`fund_adj` 得到；前复权
因请求结束日变化会改变整段历史尺度，Tushare Provider 明确拒绝，交由能满足请求的其他
Provider 降级处理。AkShare 因不回显实际方式会在复权请求中留下警告。旧版
`daily/<asset_type>/<symbol>.parquet` 可能已混合不同复权价格，升级后不自动迁移或读取；
首次更新会在新目录重新下载。由于前复权尺度会随锚定日变化，`qfq` 更新扩展到该证券全部
已有缓存日期，重取后原子替换整个文件；响应若缺少任一原缓存日期则拒绝替换。`none` 和
尺度稳定的 `hfq` 继续按缺口增量更新。

断点续传以交易所交易日历计算期望日期，将内部缺口合并为连续区间补取。批量请求只把
实际返回的日期视为已覆盖；遗漏日期会缩小为独立请求。只有所有可用数据源明确返回空
结果，才把历史日期记录到 DuckDB `daily_coverage`，熔断或数据源故障不会污染负缓存。
行情更新最多请求到系统当天；CLI 拒绝未来 `--end`，避免对尚不存在的行情反复空拉。
全市场响应不会因“非空”就视为完整。DuckDB `market_snapshots` 保存行数、证券成员、来源、
参考数量和状态。`daily_basic` 只有在数据源同时提供按请求日过滤的独立证券主表规模且达到
配置比例时才标记完整；普通非空响应只记为 `observed`，不能用同日行情把它升级。行情仅
使用已独立验证的 `daily_basic`、数据源证券主表、近期完整快照或配置下限，避免两个同时截断
的接口循环互证。Tushare 的两类全市场响应都用 `stock_basic` 的上市、退市日期计算请求日
证券数量，因此 `market-history` 回填早期历史不会套用当前 4500 只股票的下限。强制刷新
`daily_basic` 时先验证新结果；不完整版本不会覆盖已有完整快照。
可转债全市场快照使用 `cb_basic` 的上市、退市日期计算目标日独立在市数量；不使用过旧快照
作为下限，因为集中赎回会让转债存量在短期内显著收缩。Tushare 对零成交停牌转债返回的
`open/high/low=0、close=pre_close` 占位记录会被严格识别并规范为平盘 OHLC。

市场快照创建完成标记时先确认全部成员文件存在并保存 size/mtime 指纹，再按
`market_snapshot_validation_sample_size` 做确定性内容抽检。后续热路径只比较指纹，并按
`market_snapshot_cache_ttl_seconds` 在进程内限时记忆已验证快照；旧数据库缺少指纹时只
深检一次并补写。受管理的日线追加会同步刷新相关
指纹；为覆盖“写入者错误但指纹也被刷新”的剩余风险，`qt data verify` 对指定区间所有成员
做全量键与 OHLCV 审计，并同步审计 `daily_basic` 的日期、重复键和正市值；默认撤销失败
快照的完整标记。`market-history` 以 `providers.market_history_batch_days` 为界累计多个交易日，
每批只合并重写一次各证券文件，随后逐日创建完成标记。强制刷新得到残缺响应时，不覆盖同日已有的
完整标记和成员清单。任一必需数据集不完整时，微盘股信号、回测和启用该策略的每日流水线
均失败退出，不允许 join 静默缩小股票池。微盘股运行还会先验证请求区间交易日历覆盖，
再批量核对市场快照成员指纹（每个证券文件只 `stat` 一次）并验证 `daily_basic`，因此完全
缺失的一天不会因没有行情行而被遗漏。

### 3.2 分钟文件

同时支持 ZIP/CSV 和现有分钟目录。导入后的目录不再保留“股票/ETF/指数”的物理层级，
资产类型作为字段及 DuckDB 元数据保存：

```text
data/processed/minute/
  frequency=5min/
    symbol=000001_SZ/
      year=2025.parquet
```

源文件只读，输出使用 ZSTD Parquet。导入以文件状态和哈希断点续传，并记录输入行数、
过滤行数、时间覆盖和分区路径。空文件、仅表头文件或过滤后为空的文件只告警并记录，
不会删除该证券已有分区。5分钟数据可以向上聚合，但不能还原真实1分钟数据。
内部统一使用 bar 结束时间；`minute.timestamp_convention=bar_start` 会按导入频率平移时间，
`source`/`bar_end` 保留源值。`qt data minute verify` 深检分区目录元数据、唯一键、年份、
起止时间及 OHLCV 合法性。
ZIP 导入必须传入实际频率；无法解析的行、倒序/重复记录和不符合频率网格的时间点都会使
整包失败。数据先写 staging，与已有 symbol/year 分区做无冲突增量合并，再通过统一目录和
`minute_partitions` 批量提交。文件操作前写入恢复 journal，DuckDB 事务同时提交目录状态；
进程异常退出后启动时自动回滚未提交文件或清理已提交备份。分钟恢复、增量合并和提交由
跨进程文件锁串行化，避免两个导入进程交叉覆盖目录与目录表。目录导入把单个源文件视为
一个证券的完整镜像，因此发现重复证券文件会在写入前拒绝；无 manifest、无分类目录的
扁平目录按代码规则推断股票、ETF和指数。目录断点跳过前会核对登记的年份、行数、时间
覆盖和分区文件指纹。ZIP 的每个归档成员则绑定统一目录记录及分区文件指纹；增量合并后
原子刷新所有涉及该分区的历史归档指纹，避免重叠归档永久失去断点续传能力。ZIP 在
解析 CSV 前的坏包/编码等预检失败也进入失败记录和隔离目录。分区目录、`minute_partitions` 与成功的
目录来源/ZIP归档记录属于同一个提交事务；来源元数据失败时恢复物理分区和目录表，禁止
在锁外记录可能属于其他并发写入者的分区状态。

若 `preparing` 提交缺少 journal 或恢复目录，系统会停止启动，避免猜测性修复破坏数据。
恢复时应先停止其他写进程，核对 `.staging/minute-commit/<commit_id>/journal.json`、备份文件、
正式分区和 DuckDB `minute_commit_log`；仅在人工确认正式文件或备份归属后清理对应状态。

## 4. 工作流

### 4.1 收盘复盘

`qt daily run` 负责交易日判断、数据补齐、市场宽度计算、策略信号生成、报告落盘和运行
状态记录。策略证券按各自 `asset_type` 和 `adjustment` 下载；基准使用独立的
`benchmark_asset_type` 和 `benchmark_adjustment`。单独调试复盘时使用
`qt review close`。复盘目标日以及日、周、月、年收益使用的每个锚点日都必须有完整快照；
省略 `--as-of` 时选择最近的完整快照，而不是任意最新行情。市场宽度将零涨跌单独归入
“平盘”，不计入上涨或下跌区间。分钟 inbox 中任一文件失败时，导入命令返回非零状态，
每日流水线默认将整次运行标为失败；可通过 `minute.fail_daily_on_import_error=false` 明确
改为告警后继续。

### 4.2 策略回测

```text
标准行情 → 策略目标权重 → 次日开盘撮合 → 绩效/交易统计 → 统一报告
```

收盘产生的目标权重只能在下一根可用行情的开盘执行。回测公共配置包含初始资金、佣金、
股票卖出印花税、滑点、无风险利率和整数手 `lot_size`（默认100）；现金不足后的成交数量
仍会重新向下取整到完整手，ETF/指数策略不收取股票印花税。绩效 CAGR 使用实际日历跨度，
稀疏序列不会再把每个观测误当成日收益。策略配置中的 `benchmark` 若存在本地行情，会自动
加入比较；缺失时报告仍生成，但基准列显示为空。

持仓证券停牌时使用最近已知价格估值，不允许以缺失价格成交。目标订单因缺少开盘价无法
执行时保持待处理状态，并在恢复交易后的下一个开盘继续尝试；等待期间只重试受阻证券，
不会把组合中其他持仓逐日强制再平衡。卖单受阻导致现金不足时，依赖该现金的未完成买单
也保留；恢复交易后按先卖后买顺序补齐，零数量订单不进入成交记录。

ETF 轮动把 `rebalance_days=1/5/20-23` 明确定义为日/自然周/自然月周期，以实际行情日期
识别周期，不再用 Monday-Friday 工作日近似交易所日历；修改回测 `--start` 不会平移后续
完整周期的调仓日。其他任意天数会在配置加载时拒绝，避免节假日造成含糊语义。
微盘股的 weekly/monthly 调仓使用每周/每月首个可用交易日，不依赖输入数据中是否存在
未来日期，因此实盘截断到当天与事后重算的历史信号一致。

每次 `qt strategy backtest <name>` 在不可覆盖的
`artifacts/backtests/<name>/<YYYYMMDDTHHMMSSffffff>/` 生成：

```text
report.html             自包含的完整阅读页面，图表以内嵌数据保存
report.md               可归档、可版本比较、可人工补充的报告
equity.png              净值、基准与回撤
monthly_returns.png     月度收益分布
equity.csv              每日权益
positions.csv           每日持仓权重
trades.csv              订单成交与成本
metrics.json            策略和基准的机器可读指标
```

HTML 是主要阅读入口，把策略配置、核心绩效、年度/月度表现、交易成本、可信度说明和图表
集成在一个页面中；Markdown 与它使用同一份计算结果，二者不允许分别计算指标。

## 5. 报告可信度原则

自动报告只陈述引擎实际计算或验证的内容。当前引擎已经控制“收盘信号不能在当日成交”，
并建模佣金、股票卖出印花税、滑点和可配置整数手。以下事项尚未完整建模，必须在报告中
显式标注：

- 涨跌停和停牌导致的无法成交；
- A股 T+1；
- 财务数据公告时点；
- 退市股票和历史股票池导致的生存者偏差。

成交记录目前是订单流水，尚未配对为完整开平仓。因此单笔胜率、盈亏比、平均持有期、
盈利交易集中度不应伪造为自动指标；这些字段应在订单配对和独立实验能力完成后再加入。

## 6. Dashboard

`qt dashboard` 是统一浏览入口：复盘、策略信号、回测和数据任务分为独立标签页。回测
标签页选择最新运行并直接嵌入自包含的 `report.html`；旧版直接写在策略目录的产物以及
没有 HTML 的产物仍会回退显示指标 JSON 和权益图。DuckDB 采用单写者模型：dashboard 可并发读取，但不要同时运行多个非分钟数据
写任务。分钟导入已有专用跨进程锁；其余写任务仍建议由单一调度器串行执行。

手工以非交易日传入策略 `--as-of` 时，若本地最后一根行情不是该日，命令会拒绝输出过期
信号；应改用最近交易日或先补齐请求日数据。

## 7. 后续演进

1. 增加成交约束层，支持涨跌停和A股 T+1；
2. 将订单配对为持仓周期，补齐交易胜率、盈亏比、持有期和个股贡献；
3. 增加历史成分股/可交易股票池，控制生存者偏差；
4. 把参数扰动、样本外和成本压力测试建模为批量实验，而不是单次回测结论；
5. 增加回测运行版本对比与批量实验索引。
